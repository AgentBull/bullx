# RFC 0007: BullX I18n

- **Author**: Boris Ding
- **Created**: 2026-04-23

## 1. TL;DR

BullX I18n is the translation and localization subsystem for BullX. It must satisfy four constraints simultaneously:

1. **Rails / i18n-next–style key lookup.** Application code writes `t("users.greeting", name: "Alice")`, not `gettext("Hello %{name}")`. Keys are stable translation identifiers; message text is **data**, not a literal in code.
2. **Unicode ICU MessageFormat 2 (MF2) as the underlying message engine.** Plurals, gender, nested selectors, locale-aware formatting of numbers, dates, currencies, and units are all handled by MF2. BullX does not reimplement any of that.
3. **Single call path for Phoenix and plain Elixir servers.** `BullXWeb` controllers, LiveViews, and HEEx templates must be able to call it. `BullXGateway` adapters (plain Elixir, no Phoenix dependency) and the future `BullXAIAgent` subsystem must also be able to call it.
4. **Locale is application-global for now, not per-user.** A BullX deployment runs with exactly one runtime locale, managed through `BullX.Config` (RFC 0001). There is no `Accept-Language` negotiation, no `users.preferred_locale`, no session storage, no `/:locale/...` URL scope. Switching locales is an operator action (change a config value), not a user action (pick from a dropdown). Per-user multilingual support is deferred to a later RFC; the present design does not block it.

Given these constraints the design is:

- A Phoenix-independent core `BullX.I18n` that owns the catalog, resolver, and public API.
- A thin Phoenix adapter `BullXWeb.I18n` (HEEx helpers + Ecto error translator) layered on top.
- TOML files under `priv/locales/` as the on-disk translation format, keyed by BCP 47 locale.
- `elixir-localize` for MF2 parsing, formatting, CLDR locale validation, and per-process locale storage.
- Removal of Phoenix's generated `BullXWeb.Gettext` backend and the `gettext` dependency.

### 1.1 Cleanup plan

- **Dead code to delete**

  - Reuse the existing root-level infrastructure supervisor pattern (`BullX.Config.Supervisor`, `BullX.Repo`). Do not introduce a dedicated wrapper supervisor around a single `Catalog` GenServer.

  - The locale setting flows through `BullX.Config`; I18n does not maintain a parallel precedence stack.

  - `mix test test/bullx/i18n`
  - `mix i18n.check`
  - `mix precommit`

## 2. Subsystem placement

I18n is **cross-cutting infrastructure** on par with `BullX.Config` and `BullX.Repo`. It does not belong to a single L2 subsystem.

- **Namespace:** `BullX.I18n` (L1, under `lib/bullx/i18n/`).
- **Not under `BullXWeb`**, because `BullXGateway` outbound messages and future `BullXAIAgent` prompt templates also need locale-aware rendering.
- **Not under `BullXGateway` or `BullXBrain`**, because the web UI is a primary consumer.
- **OTP application:** remains the single `:bullx` application; I18n is not a separate OTP app.

Phoenix-specific helpers (HEEx helpers, Ecto changeset error translation) live under `BullXWeb.I18n` as a thin, **optional** adapter layer on top of `BullX.I18n`. The core modules have no Phoenix dependency. Because the locale is global, this adapter is thinner than a typical Phoenix i18n integration — there is no Plug, no LiveView `on_mount` hook, and no URL locale scope.

## 3. Design decisions

### 3.1 Why replace Phoenix Gettext

Gettext's msgid is the English source string itself: the translation key is **coupled** to the English text. This produces three problems:

1. Changing the English text changes the key. All translations in all languages must be re-pegged; Git history shows a string diff rather than a semantic change.
2. Plural rules are expressed via `msgid_plural` and do not support multi-selector cases (for example gender × count). Nested selection requires string concatenation.
3. `.po` files only express a flat msgid → msgstr mapping and have no structured namespacing.

The Rails / i18n-next convention decouples the key (`users.greeting`) from the message (`"Hello, {$name}"`). Namespaces nest naturally; keys are stable; translator workflows and CI checks are simpler.

### 3.2 Why elixir-localize rather than a hand-rolled MF2

`Localize.Message` already implements the full MF2 specification (simple and complex messages, `.input` / `.local` declarations, `.match` multi-selectors, CLDR-backed plugin functions `:number` / `:date` / `:currency` / `:unit` / `:list`), ships with CLDR v48.2 data, and provides:

- **Per-process locale:** `Localize.put_locale/1`, `Localize.get_locale/0`, `Localize.with_locale/2`. Decoupled from any web framework.
- **`~M` sigil and `mix format` plugin:** MF2 source is validated and canonicalized at compile time, so reformatting does not silently change keys.
- **Extensible function registry:** `config :localize, :mf2_functions` for BullX-specific formatters such as `:username` or `:signal_ref`.

BullX reuses the MF2 engine and locale semantics and **adds only what Localize does not provide**: loading per-key translation dictionaries from disk.

### 3.3 Why TOML rather than YAML, JSON, or PO

- **Multi-line strings are required for MF2.** `.match` messages regularly span several lines. TOML 1.1's `'''...'''` literal multi-line strings perform no escaping, which matches MF2's `{`, `}`, `|`, and `\` characters exactly. Translators do not need to escape anything.
- **Nested tables map to key paths.** A `[users.profile]` table is the set of `users.profile.*` keys; the file structure is the key tree.
- **Strong typing and explicit tables.** Unlike YAML's tab/space/indent semantics, TOML is rule-light and unambiguous. Missing a space after a colon does not restructure a file.
- **Tooling.** Cargo, Hex itself, and parts of the npm ecosystem already use TOML. Editor support is mature; translators do not need to learn YAML's implicit type coercion.

The parser is [`toml_elixir ~> 3.0`](https://github.com/nikolauska/toml_elixir) with `spec: :"1.1.0"` to enable multi-line strings and trailing array commas.

### 3.4 File format conventions

- Directory: `priv/locales/`.
- File name: `<BCP-47>.toml`, for example `en-US.toml`, `zh-Hans-CN.toml`, `zh-Hant-TW.toml`, `ja-JP.toml`.
- The root table is the translation tree. Meta information lives under the reserved `[__meta__]` table.
- Load order is lexicographic by file name; later files override earlier ones. (Equivalent-locale sharding is permitted but rarely used.)

This aligns with `Localize`'s supported-locale model: the locale list derived from `priv/locales/*.toml` is pushed by `BullX.I18n.Catalog` during boot, after the files have been scanned.

## 4. Translation resource layout

### 4.1 Directory structure

```text
priv/locales/
├── en-US.toml          # source language (translation source of truth)
├── zh-Hans-CN.toml     # Simplified Chinese
├── zh-Hant-TW.toml     # Traditional Chinese (Taiwan)
├── ja-JP.toml
└── ...
```

`en-US` is the source language by convention. The key set of every other locale must be a **subset** of `en-US`'s key set; `mix i18n.check` enforces this in CI (§11.2). This makes the "English first, translate after" workflow explicit.

### 4.2 TOML schema

The top-level table is the first segment of a translation key; nested tables correspond to deeper key paths. Leaves must be a string (an MF2 message) or one of the following richer forms:

1. **Plain string leaf:** `key = "MF2 message"`.
2. **Multi-line string leaf:** `key = '''multi-line MF2'''`.
3. **Rich-leaf table:** explicitly marked with `__mf2__ = true` and containing `message`, `description`, and `placeholders` fields (for translator or LLM collaboration context).

`__meta__` is a reserved top-level table and does not become part of the translation tree:

```toml
[__meta__]
bcp47 = "zh-Hans-CN"
fallback = "en-US"       # optional cross-locale fallback; must name a loaded locale
revision = "2026-04-23"  # optional, for release pipeline use
```

### 4.3 Example

`priv/locales/en-US.toml`:

```toml
[__meta__]
bcp47 = "en-US"

[users]
greeting = "Hello, {$name}!"

[users.profile]
title = "Profile"
email_label = "Email address"

# Multi-line MF2 with .match — pluralization and empty-state handling.
[users.cart]
summary = '''
.input {$count :integer}
.match $count
  0 {{Your cart is empty.}}
  1 {{You have one item in your cart.}}
  * {{You have {$count} items in your cart.}}
'''

[users.cart.checkout_button]
__mf2__      = true
message      = "Checkout {$total :currency currency=$currency}"
description  = "Primary CTA on the cart page. {$currency} is an ISO 4217 code."
placeholders = { total = "decimal amount", currency = "ISO 4217 code, e.g. USD" }

[errors.not_found]
title = "Not found"
body  = "We could not find {$resource}."
```

`priv/locales/zh-Hans-CN.toml`:

```toml
[__meta__]
bcp47 = "zh-Hans-CN"
fallback = "en-US"

[users]
greeting = "你好，{$name}！"

[users.profile]
title = "个人资料"
email_label = "电子邮箱"

[users.cart]
summary = '''
.input {$count :integer}
.match $count
  0 {{购物车是空的。}}
  1 {{购物车里有 1 件商品。}}
  * {{购物车里有 {$count} 件商品。}}
'''

[users.cart.checkout_button]
__mf2__ = true
message = "结算 {$total :currency currency=$currency}"

[errors.not_found]
title = "未找到"
body  = "抱歉，找不到 {$resource}。"
```

### 4.4 Fallback chain

Given a locale, for example `:"zh-Hans-CN"`, the fallback chain is constructed and deduplicated in the following order, returning the first hit:

1. The requested locale (`zh-Hans-CN`).
2. Any loaded CLDR parent locales for the request, from most specific to least specific (`zh-Hans-CN → zh-Hans → zh`).
3. The locale's `__meta__.fallback`, if present and if it names a loaded locale (for example `en-US`).
4. Any loaded CLDR parent locales for that declared fallback.
5. The application default locale from `BullX.Config.I18n.i18n_default_locale!/0` (§5.3), if it is not already present.
6. Any loaded CLDR parent locales for the application default locale.
7. `:"en-US"` as the final backstop, so the source language always resolves.

Each step is filtered against the loaded locale set from `priv/locales/*.toml`; BullX does not insert unsupported tags such as `:en` unless an actual TOML file exists for that locale. The key correction is that same-language parent locales are tried **before** any cross-language fallback, so `zh-Hans` can satisfy a missing `zh-Hans-CN` key before BullX jumps to English.

When no chain entry matches, §6.4 applies.

## 5. Public API

All public API lives under `BullX.I18n` and has no Phoenix dependency. Phoenix-specific helpers are described in §8.

### 5.1 Translation lookup

```elixir
@spec t(key, bindings, opts) :: String.t()
@spec translate(key, bindings, opts) :: {:ok, String.t()} | {:error, Exception.t()}

# Dotted-string keys:
BullX.I18n.t("users.greeting", %{name: "Alice"})
#=> "Hello, Alice!"

# Explicit locale (does not change the process locale):
BullX.I18n.t("users.greeting", %{name: "Alice"}, locale: :"zh-Hans-CN")
#=> "你好，Alice！"

# Scoped lookup:
BullX.I18n.t("title", %{}, scope: "users.profile")
# Equivalent to BullX.I18n.t("users.profile.title")

# Plurals and selectors: callers don't use a specialized API.
# The relevant data is passed as bindings; MF2 .match handles the branches in TOML.
BullX.I18n.t("users.cart.summary", %{count: 3})
#=> "You have 3 items in your cart."
```

The signature differs slightly from Rails. Rails mixes locale / scope with bindings in a single keyword list (`I18n.t(:key, name: "Alice", locale: :zh)`). BullX splits **MF2 bindings** from **i18n options** explicitly — bindings are the second positional argument (map or keyword list; MF2 accepts both), options are the third positional argument as a keyword list. This:

1. Prevents `:locale` / `:scope` from being interpreted as MF2 variables.
2. Produces more precise Dialyzer signatures.
3. Keeps compile-time binding validation from misreporting i18n reserved keys as missing variables.

`t/3` returns a string and degrades per §6.4 on failure. `translate/3` returns `{:ok, _}` / `{:error, _}` for callers that need to distinguish "degraded" from "successful": a Gateway adapter that receives `{:error, _}` can choose **not to send** the message to a user channel rather than sending a literal `«key»`.

### 5.2 Locale lifecycle

**Primary path: the global locale is configured through `BullX.Config` and loaded by the Catalog at boot.** Application code only calls `BullX.I18n.t/3`; it does not know or care where the locale came from. The secondary paths `put_locale/1` and `with_locale/2` are retained but intentionally used sparingly (§7.4); they serve tests, debugging, and offline rendering.

```elixir
# —— Primary path ——
BullX.Config.put("bullx.i18n_default_locale", "zh-Hans-CN")
BullX.I18n.reload()   # explicit apply, or wait for the next redeploy

# Config read path:
BullX.Config.I18n.i18n_default_locale!()
#=> "zh-Hans-CN"

# Runtime read path: application code usually doesn't call this; t/3 reads it internally.
BullX.I18n.default_locale()
#=> %Localize.LanguageTag{cldr_locale_id: :"zh-Hans-CN", ...}

# —— Secondary path (rarely used) ——
# Per-process override (tests, debugging, offline rendering):
BullX.I18n.put_locale(:"ja-JP")
BullX.I18n.with_locale(:"en-US", fn ->
  BullX.I18n.t("emails.audit_copy.body", %{...})   # audit mail always in English
end)
```

`put_locale`, `with_locale`, `get_locale`, `default_locale`, and `put_default_locale` are `defdelegate ... to: Localize`. BullX does not maintain a second per-process locale slot.

**Storage semantics and derived processes.**

- `put_default_locale/1` writes to `:persistent_term`. Every process (including `Task`s, Oban workers, and newly spawned GenServers) reads the same value.
- `put_locale/1` and `with_locale/2` write to the **process dictionary**, which is visible only inside the calling process and is not inherited by spawned children. A `Task.Supervisor.async` or Oban-spawned process does not inherit a parent's per-process override; it reads the global default instead.

This is the semantics wanted under the "locale is global" assumption: the application's steady state always reads the default; overrides affect only the single process that set them. `Localize`'s native formatter functions (`Localize.Number.to_string/2`, etc.) share this state, so they follow whatever `put_default_locale/1` has set.

### 5.3 Binding to BullX.Config

BullX.Config's final implementation is the RFC 0001 DSL that lives in `lib/`, not a registry API. The global locale is therefore declared through a normal `use BullX.Config` module and resolved through the standard PostgreSQL -> OS env -> default precedence chain.

```elixir
# lib/bullx/config/i18n.ex
defmodule BullX.Config.I18n do
  use BullX.Config

  @envdoc false
  bullx_env(:i18n_default_locale,
    type: :binary,
    default: "en-US"
  )
end
```

At boot `BullX.I18n.Catalog`:

1. Reads the string with `BullX.Config.I18n.i18n_default_locale!()` (for example `"zh-Hans-CN"`).
2. Converts it to a `%Localize.LanguageTag{}` with `Localize.validate_locale/1`. An invalid value raises and Catalog boot fails.
3. Writes the LanguageTag to `:persistent_term` via `Localize.put_default_locale/1`.

`BullX.I18n.reload/0` **only re-syncs the default locale; it does not re-scan TOML**. After an operator changes the locale value, they explicitly call `BullX.I18n.reload/0`, or wait for the next redeploy. I18n does **not** subscribe to `BullX.Config` change events: RFC 0001 does not provide a subscription API, and building one is not justified for a 3-to-5-locale, low-frequency operator action.

`reload/0` is a millisecond-scale `GenServer.call` that performs only steps 1–3. It does not re-scan TOML files or rewrite any of the `:persistent_term` dictionary entries. The dictionaries are only (re)written at boot or on a dev hot-reload event (§10). Changing the locale is therefore cheap regardless of dictionary size.

The low-level write path remains `BullX.Config.put/2`, which stores raw strings by database key, so operator updates write `"bullx.i18n_default_locale"`. This declaration maps to env var `BULLX_I18N_DEFAULT_LOCALE` under BullX.Config's existing naming rules. The set of supported locales is derived from the `priv/locales/*.toml` scan result (§7.1). There is **no** separately declared whitelist in config code: if `BullX.Config.I18n.i18n_default_locale!()` returns a locale that is not in the scan result, Catalog refuses to start and lists the locales that are actually available.

### 5.4 Missing keys and format-error handling

Behavior is fixed and not configurable:

- **Key missing in the current locale but found via the fallback chain.** Return the hit. Log at most one `Logger.warning` per query (`event: :i18n_fallback`, metadata `key`, `requested_locale`, `resolved_locale`, `domain: :i18n`). The log records the final resolved locale, not each intermediate level. This avoids "100 untranslated keys on a page × 3 fallback levels = 300 log lines."
- **Key missing at every level of the fallback chain.** `Logger.error` (`event: :i18n_missing`, `domain: :i18n`). Return the key literal as a string, e.g. `"users.foo"`.
- **MF2 formatting failure** (missing binding, illegal function argument, etc.). `Logger.error` (`event: :i18n_format_error`, `domain: :i18n`). Return the unformatted MF2 source string.

All I18n Logger calls carry `domain: :i18n` metadata. Tests can assert "no i18n event occurred" with `ExUnit.CaptureLog.capture_log(level: :warning, fn -> ... end)` filtered on `:domain`, without matching on log strings.

Translation failures never crash the application.

### 5.5 Scope macro

For modules that frequently call translation (LiveViews, controllers, Gateway adapters), `use BullX.I18n` injects a scoped `t/1,2` helper:

```elixir
defmodule BullXWeb.UserLive.Index do
  use BullX.I18n, scope: "users.index"

  def render(assigns) do
    ~H"""
    <h1><%= t("title") %></h1>            <!-- users.index.title -->
    <p><%= t("greeting", name: @user.name) %></p>
    """
  end
end
```

The macro expands to `def t(key, bindings \\ %{}, opts \\ [])` in the current module, concatenating the scope string (a compile-time constant) in front of the key and delegating to `BullX.I18n.t/3`. Rails-style relative keys (leading `.`) use the same macro: `t(".title")` inside `BullXWeb.UserLive.Index` rewrites to `BullX.I18n.t("users.index.title")`.

**Interaction with `BullXWeb.I18n.HTML`'s injected `t/1,2,3`.** HEEx templates that `import BullXWeb.I18n.HTML` receive a `t/1,2,3` that forwards to `BullX.I18n.t/3` with no scope. When a LiveView both `use BullXWeb, :live_view` (pulling in the HEEx import) and `use BullX.I18n, scope: "..."` (defining module-local `t/1,2,3`), the module-local definition wins over the import, because Elixir resolves local defs before imports. In a scoped LiveView, `t("title")` is always scope-relative — the intuitive behavior.

## 6. Runtime architecture

### 6.1 Lifecycle

```text
Boot  → Loader scans priv/locales/*.toml
      → TomlElixir.decode/2 parses each file
      → Normalizer flattens the tree to canonical form
      → MF2 syntax is validated via Localize.Message.canonical_message!/2
      → Catalog writes each locale into :persistent_term
      ↓
Runtime  BullX.I18n.t/3:
      → :persistent_term lookup → fallback chain → Localize.Message.format/3
```

### 6.2 Catalog data structure

What lands in `:persistent_term` is one entry per locale: a flat map from dotted key to canonical MF2 string.

```elixir
@type catalog :: %{
  atom() => %{
    String.t() => String.t()          # "users.profile.title" => "Profile"
  }
}
```

Leaves are stored flat rather than nested. Lookup is O(1), and because each MF2 string has already been normalized through `Localize.Message.canonical_message/2`, different whitespace/indent variants of the same `.match` message reduce to a single canonical key. A translator cannot accidentally create two keys that translate the same message and silently overwrite one another.

`__meta__` is stored separately as `%{locale => %{fallback: atom(), revision: String.t(), ...}}`.

**Write consistency.** Boot and dev reload both replace entries locale by locale via `:persistent_term.put/2`. BEAM guarantees atomicity only at the **per-key** level; across-locale puts can be observed by concurrent fallback lookups in a partial state (for example, `en-US` already new, `zh-Hans-CN` still old). This RFC deliberately does not add a larger "swap" primitive:

- At boot there are no concurrent readers — L2 subsystems have not started (§6.5).
- During dev reload there are readers, but each key is always either the old or the new canonical MF2 message (pre-validated at load time). Readers never observe a crashing value; what a translator does observe is "this key is already translated, that one is not yet" — which is exactly the progress they want to see.
- Production does not execute the reload path; dictionaries are written once at boot.

Trading across-locale atomicity for implementation simplicity and explainability is a deliberate choice.

### 6.3 Resolver lookup order

```elixir
def lookup(key, locale) do
  chain = fallback_chain(locale)        # e.g. [zh-Hans-CN, zh-Hans, zh, en-US], filtered to loaded locales
  Enum.find_value(chain, fn l ->
    case catalog |> Map.get(l, %{}) |> Map.get(key) do
      nil -> nil
      message -> {l, message}           # return resolved locale so t/3 can log fallback
    end
  end)
end
```

Each locale's catalog is a separate `:persistent_term` entry; fallback is linear across the chain. The chain itself is cached in `:persistent_term`.

### 6.4 MF2 call chain

```elixir
def t(key, bindings, opts) do
  locale = Keyword.get(opts, :locale, Localize.get_locale())
  full_key = apply_scope(key, opts)

  case lookup(full_key, locale) do
    nil ->
      Logger.error("i18n missing",
        event: :i18n_missing, key: full_key, locale: locale, domain: :i18n)
      full_key                                    # literal fallback

    {^locale, message} ->
      format_or_fallback(message, bindings, locale, full_key)

    {resolved, message} ->
      Logger.warning("i18n fallback",
        event: :i18n_fallback, key: full_key,
        requested_locale: locale, resolved_locale: resolved, domain: :i18n)
      format_or_fallback(message, bindings, resolved, full_key)
  end
end

defp format_or_fallback(message, bindings, locale, full_key) do
  case Localize.Message.format(message, bindings, locale: locale) do
    {:ok, formatted} ->
      formatted

    {:error, err} ->
      Logger.error("i18n format error",
        event: :i18n_format_error, key: full_key, locale: locale,
        reason: err, domain: :i18n)
      message                                     # return raw MF2 source
  end
end
```

- **Locale normalization.** `Localize.validate_locale/1` is already ETS-cached (~1µs); no additional cache is required.
- **Bindings normalization.** `Localize.Message.format/3` accepts atom and string keys; `%{count: 3}` works.
- **Zero-allocation path (future).** A `format_to_iolist/3` variant that returns an iolist directly to EEx can be added for LiveView-heavy paths. Not in the first version.

### 6.5 Supervision tree placement

```text
BullX.Application
└── BullX.Supervisor
    ├── BullXWeb.Telemetry
    ├── BullX.Repo
    ├── BullX.Config.Supervisor     ← RFC 0001; must boot before Catalog
    ├── BullX.I18n.Catalog          ← GenServer owning :persistent_term entries
    ├── DNSCluster
    ├── Phoenix.PubSub
    ├── BullXGateway.CoreSupervisor
    ├── BullX.Skills.Supervisor
    ├── BullXBrain.Supervisor
    ├── BullX.Runtime.Supervisor
    ├── BullXGateway.AdapterSupervisor
    └── BullXWeb.Endpoint
```

Catalog is mounted directly under the root supervisor; **no dedicated `BullX.I18n.Supervisor` wrapper**. A wrapper would not introduce a new failure boundary, and a single-child tier is strictly added weight.

The dev-only FileSystem watcher (§10) is started as an internal child Task inside Catalog's `init/1`, gated on `Mix.env() == :dev`. It does not occupy a slot in the root supervisor.

**Startup ordering:**

1. `BullX.Repo` — required by `BullX.Config.Supervisor`'s PostgreSQL cache layer.
2. `BullX.Config.Supervisor` — Catalog's `init/1` calls `BullX.Config.I18n.i18n_default_locale!()`.
3. `BullX.I18n.Catalog` — scans TOML, populates `Localize`'s `:persistent_term` entries.
4. The remaining infrastructure children (`DNSCluster`, `Phoenix.PubSub`) boot.
5. L2 business subsystems (Gateway, Brain, Web, Runtime, Skills) plus `BullXWeb.Endpoint` — any callback can call `BullX.I18n.t/3`.

Catalog completes its initial load **synchronously** in `init/1` (not through `handle_continue`). Invalid TOML, MF2 syntax errors, or a Config-supplied locale that is not in the scan result cause `init/1` to raise, which fails application boot. `BullX.Supervisor` uses the default `one_for_one` strategy, but I18n boot errors are application-level rather than transient — restarting does not help. The observable failure is `mix phx.server` exiting with an error message that includes the offending file name, line number, or locale list.

## 7. Usage from non-Phoenix subsystems

Because the locale is global (§1, §5.2), non-Phoenix servers do not need to inject a locale at process boundaries. **Application code simply calls `BullX.I18n.t/3`.** There is no need to extract a locale from signal metadata, or to wrap calls in `with_locale/2`. When `Localize.get_locale/0` has no per-process value set, it falls back to `default_locale/0`, which is fed from `BullX.Config.I18n`. Everything is globally consistent.

### 7.1 Gateway adapter example

```elixir
defmodule BullXGateway.Adapters.Feishu do
  @behaviour BullXGateway.Adapter

  def send_outbound(%{recipient: recipient, signal: signal}) do
    body = BullX.I18n.t("gateway.outbound.notification", %{
      signal_type: signal.type,
      scope_name: recipient.scope_name
    })

    FeishuOpenAPI.Message.send(recipient.channel_id, %{
      msg_type: "text",
      content: %{text: body}
    })
  end
end
```

The adapter neither imports `BullXWeb.*`, depends on Phoenix, nor reads a locale from the recipient or signal. It uses the same `BullX.I18n.t/3` as `BullXWeb.UserController`; the rendered language depends on the global locale set in `BullX.Config.I18n`.

### 7.2 Brain and async tasks

Processes spawned by `Task.Supervisor.async/2`, Oban, or similar supervisors **also** read the global locale; no cross-process locale context is required. Long-lived processes that started before an operator changed the locale (`ScopeWorker`, LiveView) automatically see the new value on the next `t/3` call — `Localize.get_locale/0` reads `:persistent_term` each time, at ~1µs per read.

### 7.3 Future BullXAIAgent

Prompt templates also go through I18n:

```toml
# priv/locales/en-US.toml
[agent.prompts.triage]
system = '''
You are a triage agent.
Current time: {$now :datetime style=long}.
'''
```

```elixir
BullXAIAgent.call(agent,
  BullX.I18n.t("agent.prompts.triage.system", %{now: DateTime.utc_now()})
)
```

The agent prompt is rendered in whichever locale the deployment is configured to. When per-user locales are added (§14), per-request overrides go through `with_locale/2`.

### 7.4 When to use `with_locale/2`

Retained but **intentionally used sparingly**. The three legitimate uses are:

1. **Tests.** ExUnit assertions of "given this locale, the output is …".
2. **Offline or asynchronous rendering.** A Chinese-language site generating English audit email copies — an operator requirement, not a user-facing request.
3. **Multi-locale side-by-side.** Debugging tools, translation coverage reports.

Business flows — request handling, signal processing, adapter sending — **must not** use `with_locale/2`. Using it implies multiple locale contexts in one process, which conflicts with the global-locale assumption and misleads readers into thinking the site already supports per-user locales.

**Pitfall: spawned children do not inherit the override.** The process dictionary does not propagate across processes (§5.2):

```elixir
BullX.I18n.with_locale(:"en-US", fn ->
  BullX.I18n.t("audit.summary", ...)              # English (parent reads the override)

  Task.Supervisor.async(BullX.TaskSup, fn ->
    BullX.I18n.t("audit.item", ...)               # global default, not en-US
  end)
end)
```

When offline rendering does need to spawn a Task inside the block, the locale must be captured and reinstalled explicitly:

```elixir
BullX.I18n.with_locale(:"en-US", fn ->
  locale = BullX.I18n.get_locale()
  Task.Supervisor.async(BullX.TaskSup, fn ->
    BullX.I18n.with_locale(locale, fn -> BullX.I18n.t("audit.item", ...) end)
  end)
end)
```

Purely sequential code (no spawned Task / Agent / GenServer) is unaffected.

## 8. Phoenix integration

Because the locale is global, the Phoenix layer does very little. There is **no Plug negotiation, no LiveView `on_mount` hook, no URL locale scope, no session storage**. Template-level `t(...)` reads the global locale; it does not depend on the controller or LiveView putting a `locale` into assigns.

`lib/bullx_web/i18n/` contains only two files.

### 8.1 HEEx / EEx helpers (`BullXWeb.I18n.HTML`)

Injected into `BullXWeb.html/0`:

```elixir
defp html do
  quote do
    use Phoenix.Component
    import BullXWeb.I18n.HTML, only: [t: 1, t: 2, t: 3]
    # ... existing imports
  end
end
```

`t/1,2,3` forwards directly to `BullX.I18n.t/3` and returns a plain string. `Phoenix.HTML`'s safe-list machinery handles escaping; templates write `<%= t("...") %>` as usual.

**Translations do not embed markup.** Links, bold, and other structural HTML stay in HEEx; translations carry text fragments only:

```heex
<p>
  <%= t("pages.home.welcome", name: @user.name) %>
  <.link navigate={~p"/dashboard"}><%= t("pages.home.cta") %></.link>
</p>
```

Translators focus on copy and never need MF2 markup (`{#link ...}`). If a real-world case of "an indivisible paragraph with embedded markup" ever emerges, `Localize.Message.format_to_safe_list/3` is available; this RFC does not add that infrastructure preemptively.

### 8.2 Ecto changeset error translation (`BullXWeb.I18n.ErrorTranslator`)

Phoenix's current error path is the standard LiveView `CoreComponents` pattern: `input/1` filters `field.errors` and maps them through a local `translate_error/1`. BullX therefore does **not** configure a separate `BullXWeb.ErrorHelpers` hook. Instead, `BullXWeb.CoreComponents.translate_error/1` becomes a thin delegation layer:

```elixir
# lib/bullx_web/components/core_components.ex
def translate_error(error) do
  BullXWeb.I18n.ErrorTranslator.translate_error(error)
end
```

**Key selection.** Ecto changeset errors have the shape `{msg, opts}` where `opts` usually contains a `:validation` atom and some validations also carry a `:kind` or `:type`. The ErrorTranslator picks a key in this order:

1. If `opts[:validation] == :length` and `opts[:kind]` is present: normalize `opts[:type]` to `:string`, `:binary`, or `:collection` (`:list` / `:map`) and use `errors.validation.length.<type>.<kind>`.
2. Otherwise, if `opts[:validation]` is present and `opts[:kind]` is present: `errors.validation.<validation>.<kind>`.
3. Otherwise, if `opts[:validation]` is present: `errors.validation.<validation>`.
4. Otherwise (custom error with only `msg`): fall back to `errors.<msg>` (underscore-normalized). Missing keys take the §5.4 path.

Coverage for Ecto's built-in validations (the TOML starter skeleton):

| `validation:` | `kind:` | Key |
| --- | --- | --- |
| `:required` | — | `errors.validation.required` |
| `:cast` | — | `errors.validation.cast` |
| `:format` | — | `errors.validation.format` |
| `:inclusion` | — | `errors.validation.inclusion` |
| `:exclusion` | — | `errors.validation.exclusion` |
| `:subset` | — | `errors.validation.subset` |
| `:confirmation` | — | `errors.validation.confirmation` |
| `:acceptance` | — | `errors.validation.acceptance` |
| `:length` | `:min` / `:max` / `:is`, `type: :string` | `errors.validation.length.string.<kind>` |
| `:length` | `:min` / `:max` / `:is`, `type: :binary` | `errors.validation.length.binary.<kind>` |
| `:length` | `:min` / `:max` / `:is`, `type: :list` / `:map` | `errors.validation.length.collection.<kind>` |
| `:number` | `:less_than` / `:greater_than` / `:less_than_or_equal_to` / `:greater_than_or_equal_to` / `:equal_to` / `:not_equal_to` | `errors.validation.number.<kind>` |

Other keys on `opts` (`count`, `number`, `enum`, …) are passed as MF2 bindings to `BullX.I18n.t/3`. For example:

```elixir
# changeset: {:too_short, [count: 3, validation: :length, kind: :min, type: :string]}
BullX.I18n.t("errors.validation.length.string.min", %{count: 3})

# changeset: {:required, [validation: :required]}
BullX.I18n.t("errors.validation.required", %{})

# changeset: {"must be less than %{number}", [validation: :number, kind: :less_than, number: 2]}
BullX.I18n.t("errors.validation.number.less_than", %{number: 2})
```

Starter TOML (`en-US`):

```toml
[errors.validation]
required     = "can't be blank"
cast         = "is invalid"
format       = "has invalid format"
inclusion    = "is invalid"
exclusion    = "is reserved"
subset       = "has an invalid entry"
confirmation = "does not match confirmation"
acceptance   = "must be accepted"

[errors.validation.length.string]
min = '''
.input {$count :integer}
.match $count
  1 {{should be at least 1 character}}
  * {{should be at least {$count} characters}}
'''
max = '''
.input {$count :integer}
.match $count
  1 {{should be at most 1 character}}
  * {{should be at most {$count} characters}}
'''
is = '''
.input {$count :integer}
.match $count
  1 {{should be 1 character}}
  * {{should be {$count} characters}}
'''

[errors.validation.length.binary]
min = '''
.input {$count :integer}
.match $count
  1 {{should be at least 1 byte}}
  * {{should be at least {$count} bytes}}
'''
max = '''
.input {$count :integer}
.match $count
  1 {{should be at most 1 byte}}
  * {{should be at most {$count} bytes}}
'''
is = '''
.input {$count :integer}
.match $count
  1 {{should be 1 byte}}
  * {{should be {$count} bytes}}
'''

[errors.validation.length.collection]
min = '''
.input {$count :integer}
.match $count
  1 {{should have at least 1 item}}
  * {{should have at least {$count} items}}
'''
max = '''
.input {$count :integer}
.match $count
  1 {{should have at most 1 item}}
  * {{should have at most {$count} items}}
'''
is = '''
.input {$count :integer}
.match $count
  1 {{should have 1 item}}
  * {{should have {$count} items}}
'''

[errors.validation.number]
less_than                = "must be less than {$number}"
greater_than             = "must be greater than {$number}"
less_than_or_equal_to    = "must be less than or equal to {$number}"
greater_than_or_equal_to = "must be greater than or equal to {$number}"
equal_to                 = "must be equal to {$number}"
not_equal_to             = "must be not equal to {$number}"
```

Third-party validations (Ash, custom `validate/2`, etc.) that supply their own `:validation` atom automatically take rule 2 or 3 depending on whether they provide `:kind`; authors only need to add matching `errors.validation.<atom>` keys.

### 8.3 LiveView

**No special integration required.** A LiveView process reads the global locale through `Localize.get_locale()` → `:persistent_term`; it is indistinguishable from a controller call. No `on_mount` hook. No `socket.assigns.locale` (templates call `t/3` directly rather than reading locale from assigns).

When an operator changes the locale and invokes `BullX.I18n.reload/0`, the next LiveView re-render (`handle_event` / `handle_info` / `push_patch`) automatically uses the new locale. Force-refreshing all open connections requires application-level coordination (PubSub broadcast plus `push_patch`) and is out of scope for I18n infrastructure.

## 9. Development hot reload

`Catalog.init/1` inspects `Mix.env/0`; in `:dev`, it starts an internal `FileSystem`-backed Task that watches `priv/locales/`:

- **`:modified`** — reload the file and atomically replace that locale's `:persistent_term` entry.
- **`:created`** (new TOML file) — load the file, add the new entry, rescan the directory, and feed the updated list to `Localize.put_supported_locales/1`.
- **`:removed`** — remove the corresponding `:persistent_term` entry and re-push `supported_locales` as above.
- **TOML syntax error or MF2 syntax error** — `Logger.error`, **retain the previous dictionary**. A typo must not crash LiveView for the whole team.
- Optionally, reuse `Phoenix.CodeReloader`'s `listeners` (already configured in `mix.exs`) to trigger the scan instead of running a second FileSystem subscription.

In `:test`, BullX does **not** start a background watcher. Tests that rewrite locale fixtures call an explicit reload function (for example `BullX.I18n.Catalog.reload_locales!/0`) after changing files; ExUnit should not carry an always-on filesystem process just to exercise reload semantics. A full reload also drops previously loaded locales that are no longer present on disk so synthetic test catalogs cannot leak through `:persistent_term`.

Production does not start the reloader; all locales are loaded once at boot.

## 10. Error handling and Mix tasks

### 10.1 Error taxonomy

| Error | Trigger | Handling |
| --- | --- | --- |
| MF2 syntax error (boot) | translator typo | Catalog `init/1` raises, application boot fails, error carries file name + line number |
| MF2 syntax error (dev reload) | translator typo | `Logger.error` + keep the previous catalog |
| Key missing at every fallback level | application code and TOML drifted apart | §5.4: `Logger.error` + return key literal |
| Missing MF2 binding or format function error (runtime) | application code forgot / passed wrong value | §5.4: `Logger.error` + return raw MF2 source |

I18n defines no exception types of its own. Boot-time load errors are `Localize.Message` exceptions bubbled up with position information; runtime missing-translation and format errors go to `Logger` rather than raising. Tests assert "no `:i18n_missing` event" with `ExUnit.CaptureLog` — no `rescue` clauses required.

### 10.2 Mix tasks

- `mix i18n.check` — offline sanity check. Confirms every non-source locale's key set is a subset of `en-US.toml`, and that MF2 variables (`{$name}`, etc.) match the source for every translated key. Added to the `precommit` alias and to CI.

`mix i18n.extract` and `mix i18n.stats` have value but are deferred: add them when the volume of translations makes manual tracking impractical.

## 11. Dependencies and configuration

### 11.1 `mix.exs`

```elixir
defp deps do
  [
    # ...existing...
    {:localize, "~> 0.1.0"},
    {:toml_elixir, "~> 3.0"},
    {:file_system, "~> 1.0", only: :dev},   # already pulled in indirectly by phoenix_live_reload
    # removed: {:gettext, "~> 1.0"}
  ]
end
```

### 11.2 `config/config.exs`

```elixir
config :localize,
  # Catalog scans priv/locales/*.toml at boot and calls Localize.put_supported_locales/1.
  # Before Catalog is up, keep Localize on its CLDR-backed defaults so MF2 formatters
  # can still resolve locale data.
  default_locale: :en,
  mf2_functions: %{
    # Reserved for BullX-specific MF2 formatters (§3.2)
  }

config :bullx, :i18n,
  locales_dir: "priv/locales"
```

The TOML directory is boot/static configuration, but subsystem code still reads it through `BullX.Config.I18n`; `config :bullx, :i18n` is only the application-config binding source. The global locale is also owned by `BullX.Config.I18n` (§5.3):

```elixir
# lib/bullx/config/i18n.ex
defmodule BullX.Config.I18n do
  use BullX.Config

  @envdoc false
  bullx_env(:i18n_default_locale,
    type: :binary,
    default: "en-US"
  )

  @envdoc false
  bullx_env(:i18n_locales_dir,
    key: [:i18n, :locales_dir],
    type: :binary,
    default: "priv/locales"
  )
end
```

These declarations map to database keys such as `"bullx.i18n_default_locale"` and env vars such as `BULLX_I18N_DEFAULT_LOCALE` under BullX.Config's existing naming rules. `i18n_locales_dir` additionally reads the boot/static `config :bullx, :i18n, locales_dir: ...` value through `BullX.Config.ApplicationBinding`.

**No separate whitelist.** The set of legal locales is the `priv/locales/*.toml` scan result. If `BullX.Config.I18n.i18n_default_locale!()` returns a locale that is not in that set, Catalog refuses to start and lists the available locales. A static whitelist in code would be a second source of truth that silently drifts from the filesystem.

**Do not bind `supported_locales` in `config/runtime.exs`.** A runtime-config callback cannot call `BullX.I18n.available_locales/0` because Catalog has not started yet. Catalog performs the scan in its own `init/1` and calls `Localize.put_supported_locales/1` — ordering is explicit and acyclic.

### 11.3 `.formatter.exs`

Do not configure a Localize formatter plugin. The Hex `localize` package used by BullX (`~> 0.1.0`) does not ship `Localize.Message.Formatter.Plugin`; TOML catalogs remain translator-authored source files and are validated by `mix i18n.check`.

```elixir
[
  import_deps: [:ecto, :ecto_sql, :phoenix],
  subdirectories: ["priv/*/migrations"],
  plugins: [Phoenix.LiveView.HTMLFormatter],
  inputs: [
    "*.{heex,ex,exs}",
    "{config,lib,test}/**/*.{heex,ex,exs}",
    "priv/*/seeds.exs"
  ]
]
```

MF2 multi-line strings inside TOML files are canonicalized by `mix i18n.check` (via `Localize.Message.canonical_message/2`) for drift reporting only; the task does not rewrite translator-authored TOML files, since authors have legitimate layout preferences.

## 12. Relationship to Localize and Gettext

### 12.1 Responsibility split

| Responsibility | Owner |
| --- | --- |
| MF2 parsing and formatting | Localize |
| CLDR locale validation and fallback chain construction | Localize |
| Locale-aware number, date, currency, unit formatting | Localize (MF2 built-in functions) |
| Per-process locale storage | Localize |
| **Per-key translation dictionary lookup** | **BullX.I18n** |
| **Loading TOML translation files from disk** | **BullX.I18n** |
| **Binding the application-global locale to `BullX.Config.I18n` and `Localize.put_default_locale/1`** | **BullX.I18n** |
| **Catalog lifecycle and dev hot reload** | **BullX.I18n** |
| HEEx `t` helper, Ecto changeset error translation | BullXWeb.I18n |

`BullX.I18n` does not wrap Localize tightly. Application code that needs `Localize.Number.to_string/2` calls it directly without going through I18n. I18n is narrowly "look up a message by key and let Localize render it" plus "bind the global locale through `BullX.Config.I18n`".

### 12.2 Removing the generated Gettext

- Delete `lib/bullx_web/gettext.ex`.
- Delete `priv/gettext/` (entire directory).
- Remove `{:gettext, ...}` from `mix.exs`.
- Remove `import BullXWeb.Gettext` (including `use Gettext, backend: BullXWeb.Gettext`) from `BullXWeb`'s `use` expansion in `lib/bullx_web.ex`.
- Replace the current Gettext-based implementation of `BullXWeb.CoreComponents.translate_error/1` with a delegation to `BullXWeb.I18n.ErrorTranslator`, which looks up `errors.validation.<atom>` per §8.2.

### 12.3 Localize's internal Gettext

Localize ships an internal `Localize.Gettext` backend that is used **only to localize Localize's own error messages** (for example the text of `%Localize.UnknownLocaleError{}`). It has nothing to do with application translations; BullX does not touch it. When Localize emits an error message, it is automatically rendered in the current locale. This is one reason I18n cannot boot before Localize: Localize's own error messages also need bootstrapping.

## 14. Out of scope

The following are deferred and do not have concrete designs in this RFC:

1. **Per-user locale (the most important future extension).** The current design's hook points for it are preserved: `put_locale` / `with_locale` are already delegated to Localize at zero cost, plus a future `BullXWeb.I18n.UserLocalePlug`, LiveView `on_mount` hook, and per-outbound `with_locale(recipient.locale, fn -> ... end)` in Gateway adapters. Schema fields, Accept-Language negotiation, and URL scope conventions belong to a dedicated follow-up RFC.
2. **Translation editor.** A web UI for non-technical translators requires AuthN, review workflow, and a publishing pipeline. Not covered.
3. **`mix i18n.extract` / `mix i18n.stats` / LLM-assisted translation.** Added when manual coverage tracking becomes impractical.
4. **SMS and email templates.** `BullXMailer` templates going through I18n is an implicit assumption; MJML / `phoenix_swoosh` integration is handled in the mailer RFC.
5. **RTL and bidi layout.** Arabic, Hebrew UI mirroring is a `BullXWeb` CSS concern. This RFC only guarantees that MF2 bidi control characters are handled per spec (already provided by Localize).
6. **Ash compatibility.** If a subsystem adopts Ash, `BullX.I18n` is still called with the same API; an Ash changeset error bridge will need a thin adapter, but the I18n core is unchanged.

## 15. Acceptance criteria

This RFC is complete when all of the following hold:

1. `mix deps.get` adds `:localize` and `:toml_elixir`, removes `:gettext`, and does not duplicate or downgrade existing dependencies.
2. `lib/bullx_web/gettext.ex`, `priv/gettext/`, and the `use Gettext` expansion inside `lib/bullx_web.ex` are deleted.
3. `BullX.I18n.Catalog` is mounted under `BullX.Supervisor` between `BullX.Config.Supervisor` and the L2 subsystems.
4. `lib/bullx/config/i18n.ex` exists and defines `BullX.Config.I18n.i18n_default_locale!/0` and `i18n_locales_dir!/0` via `bullx_env(...)`.
5. `priv/locales/en-US.toml` and `priv/locales/zh-Hans-CN.toml` exist and parse successfully. `en-US.toml` is the source-language superset.
6. `BullX.I18n.t("users.greeting", %{name: "Alice"})` returns the correct value under each configured locale, including the default and under `with_locale/2`.
7. `BullX.I18n.t/3` of a non-existent key returns the key literal and emits `event: :i18n_missing`.
8. `BullX.I18n.t/3` with a missing MF2 binding returns the raw MF2 source and emits `event: :i18n_format_error`.
9. A fallback-chain hit returns the resolved message and emits exactly one `event: :i18n_fallback` log entry with `requested_locale` and `resolved_locale` metadata.
10. `BullXWeb.I18n.ErrorTranslator` translates Ecto changeset errors through the TOML starter skeleton in §8.2.
11. HEEx templates use `t(...)` from `BullXWeb.I18n.HTML`; no module imports `BullXWeb.Gettext`.
12. In `:dev`, editing a TOML file under `priv/locales/` hot-reloads the affected locale without restarting the application, and test coverage exercises reload through an explicit API rather than a background watcher.
13. `mix i18n.check` passes and is wired into the `precommit` alias.
14. `mix compile --warnings-as-errors`, `mix format --check-formatted`, and `mix precommit` all succeed.
