# RFC 0005: UUID Primary Keys Must Be App-Generated UUIDv7

- **Author**: Codex
- **Created**: 2026-04-23

## 1. TL;DR

BullX currently has only three PostgreSQL-backed schemas under `lib/`, and none of their primary keys are UUIDs:

- `app_configs.key` is `text`
- `gateway_dedupe_seen.dedupe_key` is `text`
- `gateway_dead_letters.dispatch_id` is `text`

So there is no current runtime bug where PostgreSQL is auto-generating UUID primary keys for inserts from `lib/`. The gap is that the repository does not yet encode the project rule for future UUID-backed tables.

This RFC standardizes that rule in two places:

1. `AGENTS.md` documents the repository rule: if a PostgreSQL table primary key is UUID, BullX code must generate the value with `BullX.Ext.gen_uuid_v7/0` before insert instead of relying on a database default.
2. `lib/` gains a reusable Ecto UUID type that autogenerates UUIDv7 through `BullX.Ext.gen_uuid_v7/0`, and tests audit the codebase for forbidden UUID primary-key patterns.

## 2. Scope

### 2.1 In scope

- Add a small Ecto type dedicated to UUIDv7 primary-key autogeneration.
- Add tests that enforce the UUID-primary-key rule across `lib/` schemas and PostgreSQL migrations.
- Add the rule to `AGENTS.md` as a standardized project guideline.

### 2.2 Out of scope

- Changing existing PostgreSQL schemas or migrations that do not use UUID primary keys.
- Introducing a new repo-wide schema macro or replacing existing non-UUID primary keys.
- Modifying supervision, persistence guarantees, or message flow.

## 3. Cleanup plan

- **Dead code to delete**
  - None. The codebase does not currently contain UUID-primary-key insert paths in `lib/`.
- **Duplicate logic to merge / patterns to reuse**
  - Reuse `BullX.Ext.gen_uuid_v7/0` as the single source of UUIDv7 generation.
  - Reuse `Ecto.UUID` cast/load/dump behavior rather than inventing a parallel UUID encoder.
- **Actual code paths / schemas / documents changing**
  - Add one Ecto type module under `lib/bullx/` for UUIDv7 autogeneration.
  - Add one audit-oriented test file under `test/bullx/`.
  - Add one standards bullet to `AGENTS.md`.
- **Invariant that must remain true**
  - Existing inserts into `app_configs`, `gateway_dedupe_seen`, and `gateway_dead_letters` remain behaviorally unchanged.
  - Any future UUID primary key inserted from BullX code must be application-generated via `BullX.Ext.gen_uuid_v7/0`, not by PostgreSQL defaults.
- **Verification commands**
  - `mix test test/bullx/ext_test.exs test/bullx/uuid_primary_key_audit_test.exs`
  - `mix precommit`

## 5. Acceptance criteria

1. `AGENTS.md` explicitly states the UUID primary-key rule.
2. `BullX.Ecto.UUIDv7` autogenerates canonical UUIDv7 strings through `BullX.Ext.gen_uuid_v7/0`.
3. Tests fail if a schema under `lib/` defines an autogenerated UUID primary key without using `BullX.Ecto.UUIDv7`.
4. Tests fail if a PostgreSQL migration introduces a database-side UUID primary-key default.
5. `mix precommit` passes.
